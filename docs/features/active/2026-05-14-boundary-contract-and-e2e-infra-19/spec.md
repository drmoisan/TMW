## Overview

The host (Outlook task pane / commands TypeScript) and the backend service
(`TaskMaster.Api`) communicate over an HTTP contract, but that contract is not
gated. The TypeScript API client (`src/taskpane/classifier-client.ts`) carries
hand-written wire types that can silently drift from the backend. There is no
OpenAPI snapshot committed to the repository, the PR-pipeline `stage-6-contract`
job is a no-op placeholder, and there is no end-to-end (E2E) lane. Downstream
work — notably the filing workflow (Prompt E2) — depends on contract tests and a
Playwright smoke suite that do not yet exist. The boundary contract gates and
the E2E lane must be established before that workflow exercises both.

Prompt C1 (emit the OpenAPI document) has not produced `artifacts/openapi/current.json`.
Per user decision, C1 is folded into this work: `TaskMaster.Api` already calls
`AddOpenApi()` / `MapOpenApi()`, so the document can be emitted and snapshotted
as part of this feature.

This spec records the implementation decisions settled by the orchestrator so
the atomic-planner can plan against a fixed tool selection and integration
approach. Where `issue.md` left a choice open (for example,
`openapi-typescript` or `orval`), this spec names the chosen option.

## Behavior

Stand up the host-to-service contract gates and the E2E infrastructure:

- Emit the backend OpenAPI document from `TaskMaster.Api` at build time and
  commit the snapshot to `artifacts/openapi/current.json`.
- Generate the TypeScript API client from that OpenAPI document; forbid
  hand-written types in the API client folder via an ESLint rule scoped to that
  folder.
- Run `oasdiff` in the PR pipeline against the OpenAPI document committed at the
  previous merge base; breaking changes block the PR unless the API version is
  bumped, and the failure message identifies the offending field.
- Lint the OpenAPI document with Spectral; the rule set includes "operations
  have descriptions", "responses have schemas", and "no inline anonymous
  schemas".
- Install Playwright and add a smoke E2E suite that runs against a test M365
  tenant on a label-gated CI job, using a service-principal (client-credentials)
  auth flow rather than interactive login.
- Add `tests/e2e/smoke.spec.ts` wired into the pre-merge pipeline and into a
  label-gated PR job behind the `e2e:run` label.
- Light up PR-pipeline stage 6 (contract / schema compatibility) and add a new
  pre-merge E2E smoke stage.

## Inputs / Outputs

### Inputs

- `src/TaskMaster.Api/Program.cs` and the active endpoint records
  (`ClassifyRequest`, `ClassifyResponse`, `FeedbackRequest`, `HealthResponse`)
  — source of the emitted OpenAPI document.
- `artifacts/openapi/current.json` at the PR merge base — the `oasdiff` baseline.
- `artifacts/openapi/current.json` from the PR head — the `oasdiff` revision.
- CI secrets for the E2E job: `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`,
  `AZURE_CLIENT_SECRET`, `E2E_API_BASE_URL`. These are supplied through GitHub
  secrets and must never be committed.
- PR label `e2e:run` — opt-in gate for the label-gated PR E2E job.

### Outputs

- `artifacts/openapi/current.json` — committed OpenAPI snapshot, emitted
  deterministically by `dotnet build` of `TaskMaster.Api`.
- `src/api-client/v1.ts` — committed TypeScript client types generated by
  `openapi-typescript`.
- `oasdiff` breaking-change report rendered as GitHub Actions annotations on the
  PR diff (`--format githubactions`).
- Spectral lint report (non-zero exit on error-level findings).
- Playwright smoke test results from the E2E CI jobs.

### Config keys and defaults

- `OpenApiGenerateDocuments=true`, `OpenApiDocumentsDirectory` pointing to
  `artifacts/openapi`, `OpenApiGenerateDocumentsOptions=--file-name current` in
  `TaskMaster.Api.csproj`.
- `<Version>1.0.0</Version>` in `TaskMaster.Api.csproj` — initial API version.
- `package.json` scripts: `generate:api` (client generation) and `lint:openapi`
  (Spectral lint).
- `.spectral.yaml` at repo root — ruleset extending `spectral:oas` with the
  three named error-level rules.
- `playwright.config.ts` at repo root — E2E test directory, timeouts, and
  client-credentials auth setup project.

### Versioning / backward-compatibility constraints

- The OpenAPI document carries `info.version`, sourced from
  `<Version>` in `TaskMaster.Api.csproj`. A `DocumentTransformer` is added so
  `info.version` is present in the emitted document for version-bump
  enforcement.
- A breaking contract change is permitted only when `info.version` differs
  between the merge-base snapshot and the PR-head snapshot. The CI step compares
  the two `info.version` values; if unchanged, `oasdiff breaking` failures block
  the PR.
- The emitted document must be deterministic: emit ordering and formatting are
  pinned so `oasdiff` diffs are stable across rebuilds.

## API / CLI Surface

### OpenAPI emission

- `dotnet build src/TaskMaster.Api/TaskMaster.Api.csproj` emits
  `artifacts/openapi/current.json` via `Microsoft.Extensions.ApiDescription.Server`
  (build-time). The package is a build-only dependency with `PrivateAssets=all`.
- The build-time emitter invokes app startup under the `GetDocument.Insider`
  assembly name. Auth and Graph registrations
  (`AddMicrosoftIdentityWebApi`, `AddMicrosoftGraph`) must be guarded so they do
  not run during document generation, because the `AzureAd` configuration
  section is absent in CI. Application, infrastructure, and classifier service
  registrations remain outside the guard.
- The existing `NSwag.MSBuild` package reference, the `GenerateOpenApi` MSBuild
  target, and the `EnableNSwagEmission` property group are removed; they are
  replaced by `Microsoft.Extensions.ApiDescription.Server`. The `NSwag.MSBuild`
  entry in `Directory.Packages.props` is removed if no other project references
  it.
- The `/api/ping` endpoint's anonymous response (`{ status: "pong" }`) is
  wrapped in a named response type so the emitted document contains no inline
  anonymous schema, satisfying the Spectral "no inline anonymous schemas" rule.

### TypeScript API client generation

- Tooling: `openapi-typescript` (devDependency, type generation only) plus
  `openapi-fetch` (runtime dependency, typed fetch wrapper). `orval` is not
  used.
- `npm run generate:api` runs
  `openapi-typescript artifacts/openapi/current.json --output src/api-client/v1.ts`.
- The generated client is committed at `src/api-client/v1.ts`. `src/api-client/`
  holds only generated output; no hand-written logic belongs there.
- `src/taskpane/classifier-client.ts` is migrated now: the hand-written
  `ClassifyRequest`, `ClassifyResponse`, and `FeedbackRequest` interfaces are
  replaced with imports of the generated `components["schemas"][...]` types. The
  `ClassifierClient` wrapper class and the pure helpers
  (`normalizeClassifyRequest`, `parseClassifyResponse`) are retained. A full
  rewrite of `ClassifierClient` onto an `openapi-fetch` `createClient` is
  deferred and is out of scope for issue #19.
- IE 11 / `browserslist`: this feature does not drop IE 11 from `browserslist`.
  Instead, the implementation must verify that an existing fetch polyfill covers
  `openapi-fetch`'s native-fetch usage in the supported browser targets.

### ESLint folder guard

- A new config block in `eslint.config.mjs`, scoped to `src/api-client/`, uses
  the built-in `no-restricted-syntax` rule with `error` severity to ban
  `TSInterfaceDeclaration` and `TSTypeAliasDeclaration` nodes. No custom plugin
  is added.
- The generated `v1.ts` file itself emits type aliases and interfaces; it is
  excluded from the ban scope via the `files` glob (for example
  `src/api-client/!(v1).ts`) so the guard applies only to hand-editable files.
- The existing `no-restricted-syntax` ban on non-deterministic APIs for
  `src/**/*.ts` is unchanged.

### oasdiff contract gate

- `oasdiff` runs in PR-pipeline `stage-6-contract`. The job runs on
  `ubuntu-latest`.
- Step sequence: checkout with full history, emit the PR-head document via
  `dotnet build`, extract the merge-base baseline with
  `git show origin/<base_ref>:artifacts/openapi/current.json`, then run
  `oasdiff breaking <baseline> <pr-head> --fail-on ERR --format githubactions`.
- Before invoking `oasdiff breaking`, the CI step compares `info.version`
  between the baseline and the PR-head document. If the version changed, the
  breaking-change result is treated as permitted and the step exits success. If
  the version is unchanged, error-level breaking changes fail the step.
- `--format githubactions` renders offending fields as inline annotations on the
  PR diff.

### Spectral OpenAPI lint

- Tooling: `@stoplight/spectral-cli` (devDependency).
- `.spectral.yaml` at repo root extends `spectral:oas` and defines three
  error-level rules:
  1. Operations must have a non-empty description.
  2. All responses must declare a schema.
  3. No inline anonymous schemas; request and response bodies must reference a
     named component.
- `npm run lint:openapi` runs
  `spectral lint artifacts/openapi/current.json --ruleset .spectral.yaml`.
- Spectral runs inside `stage-6-contract` after the document is emitted. A
  non-zero exit on error-level findings fails the stage.

### Playwright E2E lane

- Tooling: `@playwright/test` (devDependency); Chromium installed in CI via
  `npx playwright install --with-deps chromium`.
- `playwright.config.ts` at repo root defines the `tests/e2e/` test directory
  and a setup project that obtains an access token.
- Authentication uses the client-credentials (service-principal) flow against
  the Microsoft identity platform token endpoint. The token is stored in
  Playwright `storageState` and reused by dependent test projects. No
  interactive login is used.
- The auth setup file validates that `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`,
  `AZURE_CLIENT_SECRET`, and `E2E_API_BASE_URL` are present and throws if any are
  missing, so the job fails closed rather than skipping silently when secrets
  are absent.
- `tests/e2e/smoke.spec.ts` is the initial smoke suite covering:
  `GET /health` returns 200 with `{ status: "ok" }`; `POST /api/classify` with a
  valid bearer token and a minimal payload returns 200 with `label` and
  `confidence`; `POST /api/classify/feedback` returns 204.

### CI wiring

- `.github/actions/contract/action.yml` is replaced: the no-op placeholder
  becomes real steps that emit the document, run Spectral, and run `oasdiff`
  with the version-bump bypass check.
- `pr-pipeline.yml` gains a label-gated job `stage-e2e-smoke` that runs on
  `ubuntu-latest` and executes only when the PR carries the `e2e:run` label.
  The label gate causes the job to skip (not fail) when the label is absent.
- `pre-merge-pipeline.yml` gains `stage-10-e2e`, which runs on `ubuntu-latest`
  unconditionally after `stage-9-golden`, so E2E is verified on every merge
  attempt against `main`.
- Existing stages are not renumbered. The numbering gap created by adding
  `stage-10-e2e` is disregarded per user direction.

## Data & State

- The OpenAPI document is the new persisted contract artifact. It is emitted at
  build time and committed; it is not generated at runtime in CI from a hand
  edit. The committed snapshot is the source of truth for the `oasdiff`
  baseline.
- `src/api-client/v1.ts` is a derived artifact, committed and regenerable from
  `artifacts/openapi/current.json` via `npm run generate:api`. The generated
  file and the committed OpenAPI document must stay consistent.
- `info.version` in the OpenAPI document is the state that drives breaking-change
  enforcement; it is sourced from `<Version>` in `TaskMaster.Api.csproj`.
- No database, cache, or backfill changes are introduced. The only migration is
  the one-time replacement of hand-written wire types in
  `src/taskpane/classifier-client.ts` with generated types.
- Invariant: the committed `artifacts/openapi/current.json` must equal the
  document a clean `dotnet build` produces. CI relies on this for stable
  `oasdiff` diffs.

## Constraints & Risks

- Service-principal credentials for the test M365 tenant must be supplied
  through CI secrets, not committed; the E2E job must fail closed when secrets
  are absent.
- The OpenAPI snapshot must be deterministic so `oasdiff` diffs are stable; emit
  ordering and formatting must be pinned. The implementer must verify
  idempotency on first commit.
- New tooling (`oasdiff`, `@stoplight/spectral-cli`, `@playwright/test`,
  `openapi-typescript`, `openapi-fetch`) adds dependencies; the selected
  packages are well-maintained and widely used.
- Generated client output must satisfy the existing TypeScript toolchain gates
  (format, lint, type-check) and the 500-line file size limit.
- The pre-merge pipeline currently stops at `stage-9-golden`; the new E2E stage
  is additive and the numbering gap (no stage-10 today) is disregarded per user
  direction.
- The build-time emitter invokes app startup. If auth or Graph registrations are
  not correctly guarded against the `GetDocument.Insider` entry assembly, the
  emission build fails in CI. The startup guard is required.
- The Spectral "no inline anonymous schemas" rule will fire on the `/api/ping`
  anonymous response unless that response is wrapped in a named type; this
  feature includes that change.
- `oasdiff` has no npm distribution. The `ubuntu-latest` runner for
  `stage-6-contract` allows the official action or a downloaded Linux binary;
  the chosen install path must pin the `oasdiff` version.

## Implementation Strategy

### Implementation scope

- Add build-time OpenAPI emission to `TaskMaster.Api`, remove the NSwag emission
  path, and commit `artifacts/openapi/current.json`.
- Add a `DocumentTransformer` and `<Version>1.0.0</Version>` so `info.version`
  is present for version-bump enforcement.
- Add a `GetDocument.Insider` startup guard around auth and Graph registrations
  in `Program.cs`.
- Wrap the `/api/ping` anonymous response in a named type.
- Add `openapi-typescript`, `openapi-fetch`, and `@stoplight/spectral-cli` to
  `package.json` with `generate:api` and `lint:openapi` scripts; commit the
  generated `src/api-client/v1.ts`.
- Migrate `src/taskpane/classifier-client.ts` type declarations onto the
  generated types; keep the wrapper class and pure helpers.
- Add the `src/api-client/` ESLint folder guard in `eslint.config.mjs`.
- Add `.spectral.yaml` and `playwright.config.ts` at repo root, and the
  `tests/e2e/` directory with `smoke.spec.ts` and the client-credentials auth
  setup.
- Replace `.github/actions/contract/action.yml` with real oasdiff and Spectral
  steps including the version-bump bypass check.
- Add `stage-e2e-smoke` (label-gated) to `pr-pipeline.yml` and `stage-10-e2e`
  (unconditional) to `pre-merge-pipeline.yml`, both on `ubuntu-latest`.
- Confirm `.gitignore` does not exclude `artifacts/openapi/`.

### New or updated classes / functions / commands

- New: the `DocumentTransformer` setting `info.version`; a named response type
  for `/api/ping`; the auth setup file under `tests/e2e/`; `smoke.spec.ts`.
- Updated: `Program.cs` startup registration order and guard;
  `classifier-client.ts` type imports; `eslint.config.mjs` config blocks;
  `package.json` scripts and dependencies; the contract composite action; both
  pipeline workflow files.

### Dependency changes and rationale

- Add `Microsoft.Extensions.ApiDescription.Server` (build-time) — first-party
  build-time OpenAPI emitter that works on net10.0, replacing the blocked NSwag
  path.
- Add `openapi-typescript` (devDependency) — generates wire types from the
  OpenAPI document with zero runtime footprint.
- Add `openapi-fetch` (runtime dependency) — typed fetch wrapper imported by
  production TypeScript; chosen over `orval` because it matches the existing
  thin-wrapper pattern and is lighter.
- Add `@stoplight/spectral-cli` (devDependency) — OpenAPI linter for the
  contract stage.
- Add `@playwright/test` (devDependency) — E2E test runner.
- Remove `NSwag.MSBuild` — superseded by the build-time emitter.

### Logging / telemetry additions

- No application logging or telemetry changes. CI surfaces results through
  `oasdiff` GitHub Actions annotations, Spectral exit codes and console output,
  and Playwright test reports.

### Rollout plan

- No feature flags. The E2E PR job is gated by the `e2e:run` label, which is the
  opt-in mechanism; the pre-merge E2E stage runs unconditionally. If the
  build-time emitter proves unstable in CI, the fallback is to keep the prior
  no-op contract action disabled and address emission before re-enabling the
  gate; this is a development-time fallback only and not a shipping state for
  this issue.
